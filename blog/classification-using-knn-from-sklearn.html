¶

Stratified Train-test split¶

Our goal with training our model is to see how well it can perform on the test set or on unseen data. However, if we repeatedly evaluate the model on the test set and re-train it, we are introducing bias. We'll create a validation set in addition to train and test set. We can then train our model and evaluate it on the validation set. Depending on its performance on the validation set, we can re-train it with some tweaks and evaluate it again. Once we're satisfied with the model's performance on the validation set, we can evaluate it one last time on the test set.

The two classes are roughly balanced but we will use Stratified split to ensure equal representation of classes in both training and test sets.

We'll split that data set into three parts:
Training Set (60%)
Validation Set (20%)
Test Set (20%)

KNN for one feature

¶

k-NNs don't really have a training phase. So how would we use scikit-learn to "fit" our model? When we implemented a k-NN from scratch, we calculated the distance between observations. Across a large number of features and observations, this can be a computationally expensive task. Instead of a brute force approach, we can use different data structures to help speed that up. scikit-learn uses the training phase to set up such a data structure.

Now that we have our training and test sets, we can implement our algorithm. Before we begin, we need to select a distance metric to calculate the distance between observations. We will use Euclidean distance.

Feature Engineering (One-hot encoding)

¶

Fortunately, we don't have to rely on using just one feature at a time. We can use multiple at the same time, which might improve our model's performance. There are several categorical variables in our data set as well. Since a k-NN uses a distance metric, it can't work with string data. We can work around it without much effort.

Just like we encoded the yes and no values as 0 and 1 for our target variable, we can encode our categorical features.

The practice of dropping the first category in one-hot encoding is a common convention, particularly for linear models, to avoid perfect multicollinearity, as the sum of all one-hot encoded columns for a categorical variable equals one, creating a linear dependency. The drop='first' parameter in tools like OneHotEncoder or pd.get_dummies is designed to remove one category per feature, thereby reducing redundancy and simplifying the model. This can improve model interpretability and performance, especially in models like linear regression that are sensitive to multicollinearity.

Dropping the first category in one-hot encoding has minimal impact on the K-Nearest Neighbors (KNN) algorithm, as KNN relies on distance metrics between feature vectors and does not suffer from multicollinearity like linear models. While dropping the first category reduces dimensionality slightly—potentially improving efficiency and mitigating the curse of dimensionality—it generally does not affect model performance significantly because the information remains intact in the relative distances between samples.

In practice, for KNN, keeping all categories is often safer and more interpretable, especially since the algorithm is robust to redundant features. If dimensionality becomes an issue, techniques like PCA or feature selection are preferred over arbitrary column dropping.

Feature Selection

¶

We previously either used all the features in our dataset to build and train a model or randomly selected a handful of them. Both of them are not the best approaches to train a model.

Not all features in a dataset might be relevant to a model's performance. Identifying and removing such features in the data preparation step, before training a model, can not only boost its performance, but also reduce the computational cost. The latter is especially important when we have to work with large datasets and complex machine learning models.

There are several ways we can identify and select relevant features:

  1. Domain Expertise. For example, one of the features is euribor3m. A reasonable understanding of what Euribor is could inform of us whether it is likely to have any impact on the prediction.
  2. Identifying features that are strongly correlated to our target variable.

Methods for Feature Selection:

  1. We'll calculate the Pearson's correlation coefficient on our columns to identify which features are strongly correlated to the target variable. And we will select five features based on the highest absolute values of Pearson's r.
  2. Univariate feature selection: It works by selecting the best features based on univariate statistical tests. For classification: chi2, f_classif, mutual_info_classif can be used. We will use mutual_info_classif. It works for both continuous and categorical features and it captures both linear and non-linear relationships, making it robust and widely applicable.

It's important to note that calculating the Pearson's r is not an ideal approach. It's a measure of linear correlation between variables. Therefore, it will fail to capture any non-linear relationships. Pearson correlation assumes both variables are continuous and measures linear relationships. When the target is categorical, this assumption is violated. Additionally, the dummy variables aren't technically continuous. There are alternative approaches (such as Cramér's V) that are more suitable than Pearson's correlation coefficient when working with categorical columns.

Method 2: Using mutual_info_classif from scikit-learn¶

Estimate mutual information for a discrete target variable. Mutual information (MI) between two random variables is a non-negative value, which measures the dependency between the variables. It is equal to zero if and only if two random variables are independent, and higher values mean higher dependency.

The term “discrete features” is used instead of naming them “categorical”, because it describes the essence more accurately. For example, pixel intensities of an image are discrete features (but hardly categorical) and you will get better results if mark them as such.

Each neighbor in the above algorithm is given the same weight--no neighbor is more important or relevant than another. When creating our model, this is the same as setting the weights parameter to uniform. All neighbors have a uniform impact when our model decides on a class for the new data point.

weights parameter:
In our dataset, target classes are roughly balanced but in case of highly imbalanced dataset, we might end up with a higher likelihood of selecting the more prominent class instead of the less prominent if all neighbors were considered equally. Our model would be biased towards one class. A potential solution to that is to assign weights to those neighbors. We calculate the distance of each neighbor from the unknown data point. We assign a weight equal to the inverse of that distance. The closer the neighbor to the new point, the more likely the new point belongs to the same class.

This comes with its own potential drawbacks. For example, the additional computations will add to our computational costs. The model might also overfit the data, since it would start to learn from the closest features instead of generalizing appropriately.

KNN hyperparameter tuning using GridSearchCV

¶

Let's experiment with a few other hyperparameters and observe how they impact our model's performance. We will redo train test split because GridSearchCV performs cross validation automatically, we don't need to create a separate val set.

When performing feature scaling along with GridSearchCV, best practice is to include the scaler within a sklearn Pipeline, so that scaling is applied independently within each fold of cross-validation. This prevents data leakage and ensures that the validation data is scaled using statistics derived only from the training portion of each fold.

If scaling is performed prior to GridSearchCV, it is still valid as long as the same transformation is consistently applied, but it may not be as robust in preventing data leakage, especially in complex pipelines involving multiple preprocessing steps. Therefore, integrating scaling into the pipeline is the recommended approach for reliable cross-validation results.
